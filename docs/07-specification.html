1 システム概要

システムの目的 必須
このシステムが解決する課題と提供する価値

本システム「カフェナビ」の目的は、作業・学習向けカフェ探しと再訪管理を効率化することです。コード上では、以下の課題を解決する設計になっています。

  • 課題1: 作業しやすいカフェ情報を探しても、後で見返しにくい
    • 価値: 一覧画面でカフェを閲覧し、詳細画面で情報確認できる
  • 課題2: 気になったカフェを個人で管理しづらい
    • 価値: 会員登録・ログイン後にお気に入り登録/解除でき、再訪候補を保持できる
  • 課題3: ユーザーごとの状態管理がないと、継続利用しにくい
    • 価値: セッション管理（user_sessions）により、ログイン状態で個人データ（お気に入り）を扱える

4 機能詳細仕様

機能仕様テンプレート（コピーして使用）

機能名：新規会員登録 必須
機能ID：F-001

目的: 新規ユーザーを作成する

対応画面: signup.html

入力: userName, email, password

入力チェック: userName 2文字以上 / email に @ を含む / password 6文字以上

API: POST /api/signup

正常時: 201 Created。メッセージ表示後、ログイン画面へ遷移

異常時: 400 入力不正 / 409 メール重複。通信エラー時は画面にエラーメッセージ表示

機能名：ログイン認証 必須
機能ID：F-002

目的: ユーザー認証とログイン状態の確立

対応画面: login.html

入力: email, password

API: POST /api/login

処理: メールでユーザー取得 / パスワードハッシュ照合 / セッショントークン発行（user_sessions 保存）

正常時: 200 OK。sessionToken を localStorage 保存し、index.html に遷移

異常時: 401 認証失敗。通信エラー時メッセージ表示

機能名：ログアウト 必須
機能ID：F-003

目的: ログイン状態を終了する

対応画面: index.html（ヘッダー）

API: POST /api/logout

処理: user_sessions から該当トークン削除 / クッキー無効化 / フロントのトークン削除

正常時: 200 OK。画面を未ログイン表示に更新

機能ID：F-004

目的: カフェ情報をカード形式で一覧表示

対応画面: index.html

データ源: 画面のカードDOM（静的配置） / お気に入り状態はAPIで反映

補足: ログイン状態に応じてヘッダー表示（ログイン/ログアウト）切替

機能名：ページネーション 必須
機能ID：F-005

目的: 一覧の視認性向上

対応画面: index.html

仕様: 1ページあたり10件 / 前へ・次へ・ページ番号ボタン表示 / 表示対象件数に応じてページングの表示制御

5 非機能要件

セキュリティ要件 必須
認証・認可・データ保護に関する要件
  • 認証方式: メールアドレス + パスワード認証（/api/login）を採用する。認証成功時にセッショントークンを発行し、user_sessions テーブルで管理する。
  • 認可方式: お気に入り機能（取得・追加・削除）はログイン必須とする。未認証時は 401 Unauthorized を返却する。
  • パスワード保護: 平文保存は行わず、salt + SHA-256 ハッシュで保存する。
  • セッション管理: トークン有効期限は14日。ログアウト時はセッションを削除し、Cookieを無効化する。
  • 通信・ヘッダー: CORSヘッダーを設定し、APIアクセスを制御する。
  • データ保護: DBは外部キー制約を有効化し、参照整合性を維持する。users.email、favorites(user_id, cafe_id)、user_sessions.session_token には一意制約を設ける。
