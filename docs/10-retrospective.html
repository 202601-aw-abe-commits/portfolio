1 プロジェクト振り返り

企画・設計段階の振り返り 必須
マーケットリサーチ、ペルソナ設計、情報設計について

本プロジェクト「カフェナビ」では、まず市場調査を通して、近年のリモートワークやオンライン学習の普及により、「作業・勉強に適したカフェを探したい」というニーズが高まっていることに着目した。既存のGoogleマップや食べログなどのサービスでは、店舗数や口コミは豊富である一方、「Wi-Fiの有無」「電源席」「静かさ」「長時間利用可能か」といった作業環境に関する情報が分かりづらいという課題があった。そこで、作業環境に特化した検索サービスを制作することで差別化できると考えた。

マーケットリサーチでは、3C分析やSWOT分析、STP分析を活用し、ターゲットユーザーや競合との差別化ポイントを整理した。特に、学生やリモートワーカー、フリーランスなど、カフェを「作業場所」として利用するユーザー層に注目した点が大きな特徴である。競合サービスでは不足していた「作業環境の分かりやすさ」を強みに設定し、Wi-Fi・電源・静かさなどを条件検索できる機能を中心に設計を進めた。

ペルソナ設計では、「大学生で課題やレポート作業のためにカフェを利用する21歳の学生」という具体的な人物像を設定した。普段からスマートフォンでカフェ情報を検索し、GoogleマップやSNSを利用している行動を想定したことで、モバイルファーストなUI設計や、短時間で条件検索できる操作性を意識することができた。また、「実際に行ってみないと作業しやすいか分からない」という課題を持つことから、設備情報や作業スコアを分かりやすく表示する設計につながった。

情報設計では、「ユーザーが最短で目的のカフェを探せること」を重視した。トップページでは、検索UIをファーストビュー中央に配置し、左側に絞り込み条件、中央にカフェ一覧を配置することで、検索と比較をしやすいレイアウトにした。また、カード形式でWi-Fi・電源・営業時間などを一覧表示することで、ユーザーが直感的に判断できるよう工夫した。さらに、お気に入り機能を実装することで、気になったカフェを保存し、再訪しやすい導線も設計した。

今回の企画・設計を通して、単に見た目を作るだけではなく、「誰のどんな課題を解決するのか」を明確にすることの重要性を学んだ。また、マーケット分析やペルソナ設計を行ったことで、必要な機能やUI設計の方向性を具体化しやすくなり、ユーザー視点でサービスを考える力を身につけることができた。

デザイン段階の振り返り 必須
デザインガイドライン、ワイヤーフレーム、Figma作業について

デザイン段階では、「作業・勉強向けカフェを探しやすいサービス」というコンセプトをもとに、ユーザーが直感的に操作できるUI設計を意識して制作を進めた。特に、初めて利用するユーザーでも迷わず検索できるように、「シンプルで分かりやすい導線」を重視した。

デザインガイドラインでは、「温かみ」「シンプル」「使いやすさ」をキーワードとして設定した。カフェらしい落ち着いた雰囲気を表現するため、暖色系をベースにしたカラーパレットを採用し、長時間見ても疲れにくい配色を意識した。また、余白を十分に取り、情報を整理することで、検索条件やカフェ情報を視認しやすくした。さらに、見出し・本文・補助情報の文字サイズを分け、情報の優先度が一目で分かるようにタイポグラフィを設計した。

ワイヤーフレーム制作では、ユーザーが「最短で目的のカフェを見つけられること」を重視して画面構成を設計した。トップページでは、検索機能をファーストビュー中央に配置し、ユーザーがすぐ検索行動に移れるようにした。また、左側に絞り込み条件、中央〜右側にカフェカード一覧を配置することで、検索結果を比較しやすいレイアウトを意識した。カードにはWi-Fi・電源・営業時間・作業スコアなどをまとめて表示し、作業環境を一目で判断できるよう工夫した。さらに、お気に入りボタンをカード右上に配置することで、ユーザーが自然に保存操作を行える導線を設計した。

Figmaでのデザイン作業では、実際の画面サイズを想定しながらUI制作を進めた。PC版だけでなくスマートフォン版のレイアウトも作成し、レスポンシブ対応を前提に設計したことで、異なる端末でも操作しやすい画面構成を意識できた。また、ボタンやカード、フォームなどのUIコンポーネントを統一することで、全体に一貫性を持たせた。実際に画面を作成していく中で、「情報量が多すぎると見づらくなる」「余白によって視認性が大きく変わる」など、UI/UX設計の重要性を実感した。

今回のデザイン工程を通して、単に見た目を整えるだけではなく、「ユーザーがどう感じ、どう行動するか」を考えながら設計することの大切さを学んだ。また、ワイヤーフレームからFigmaへ落とし込む流れを経験したことで、情報設計からUIデザインまでを一貫して考える力を身につけることができた。

HTML/CSS/JavaScript の実装について

HTML/CSS/JavaScript実装では、画面遷移・状態管理・API連携の基本を一通り形にできた点が成果でした。特にJavaScriptでは、fetch を使ったログイン判定・お気に入り追加/解除・ページネーションを実装し、UI状態（ログイン時/未ログイン時）を切り替える流れまで作れました。

今後は、

バックエンド実装の振り返り 必須
Java/Spring Boot の実装について

Java/Spring Boot実装では、Controller → Service → Mapper(Repository) → DB の基本レイヤー構成を実践でき、一覧・登録・更新・削除・状態切替までの一連の処理を形にできた点が大きな成果でした。特に、Controllerを薄くしてServiceに処理を寄せる設計、RedirectAttributes を使ったユーザー向けメッセージ返却、MyBatis経由のDB操作など、実務に近い流れを経験できました。

一方で課題は、バリデーションと例外処理の標準化です。@Valid/BindingResult を使った統一的な入力検証、グローバル例外ハンドラ（@ControllerAdvice）によるエラーレスポンス統一、ログ設計（監査・障害解析向け）が不足しています。また、認証認可（Spring Security）やテスト自動化（Service/Controller層の単体・結合テスト）も今後の強化ポイントです。

今後は、

# Python（標準ライブラリ）とSQLiteで実装するカフェナビの認証・お気に入り機能

## はじめに

本記事では、Python標準ライブラリ（http.server）+ SQLite で作成した「カフェナビ」の実装をもとに、以下を解説します。

  • メール/パスワード認証
  • セッション管理
  • お気に入り登録/解除
  • フロントエンド連携（JavaScript fetch）

―――

## 1. システム概要

カフェナビは、作業向けカフェを一覧・詳細表示し、ログインユーザーがお気に入り管理できるWebアプリです。

  • フロントエンド: HTML/CSS/JavaScript
  • バックエンド: Python (server.py)
  • データベース: SQLite (db/cafe_navi.sqlite3)

―――

## 2. DB設計（主要テーブル）

  • users: ユーザー情報
  • cafes: カフェマスタ
  • favorites: ユーザー×カフェのお気に入り
  • user_sessions: セッショントークン

favorites には UNIQUE(user_id, cafe_id) を設定し、重複登録を防止しています。

CREATE TABLE IF NOT EXISTS favorites (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  cafe_id INTEGER NOT NULL,
  created_at TEXT NOT NULL,
  UNIQUE(user_id, cafe_id),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (cafe_id) REFERENCES cafes(id) ON DELETE CASCADE
);

## 3. バックエンド実装

### 3-1. パスワードのハッシュ化

平文保存はせず、salt + SHA-256 で保存/照合します。

def hash_password(password: str, salt: Optional[str] = None) -> str:
  if salt is None:
    salt = secrets.token_hex(16)
  digest = hashlib.sha256(f"{salt}:{password}".encode("utf-8")).hexdigest()
  return f"{salt}${digest}"

def verify_password(password: str, password_hash: str) -> bool:
  parts = password_hash.split("$", 1)
  if len(parts) != 2:
    return False
  salt, stored_digest = parts
  calc = hash_password(password, salt).split("$", 1)[1]
  return hmac.compare_digest(calc, stored_digest)
### 3-2. ログインAPI（セッション発行）

認証成功時にトークンを発行し、user_sessions に保存します。

if parsed.path == "/api/login":
  payload = self._read_json()
  email = str(payload.get("email", "")).strip().lower()
  password = str(payload.get("password", ""))

  user = conn.execute(
    "SELECT id, user_name, email, password_hash FROM users WHERE email = ? AND is_active = 1",
    (email,)
  ).fetchone()

  if user is None or not verify_password(password, user["password_hash"]):
    self._send_json(HTTPStatus.UNAUTHORIZED, {"error": "メールアドレスまたはパスワードが違います"})
    return

  token = secrets.token_hex(32)
  expires = datetime.now(timezone.utc) + timedelta(days=14)

  conn.execute(
    "INSERT INTO user_sessions(user_id, session_token, expires_at, created_at) VALUES(?, ?, ?, ?)",
    (user["id"], token, expires.strftime("%Y-%m-%d %H:%M:%S"), utc_now_iso())
  )
  conn.commit()

  self._send_json(HTTPStatus.OK, {"sessionToken": token, "user": {"id": user["id"], "userName": user["user_name"]}})
### 3-3. お気に入りAPI（追加/削除）

ログイン中ユーザーのみ操作可能にしています。

if parsed.path == "/api/favorites":
  user = self._require_user(conn)
  if user is None:
    return

  payload = self._read_json()
  cafe_code = str(payload.get("cafeCode", "")).strip()
  cafe_name = str(payload.get("cafeName", "")).strip() or cafe_code
  area = str(payload.get("area", "")).strip()

  conn.execute(
    """
    INSERT INTO cafes(external_code, name, area, created_at, updated_at)
    VALUES(?, ?, ?, ?, ?)
    ON CONFLICT(external_code) DO UPDATE SET name=excluded.name, area=excluded.area, updated_at=excluded.updated_at
    """,
    (cafe_code, cafe_name, area, utc_now_iso(), utc_now_iso()),
  )

## 4. フロントエンド連携

### 4-1. ログイン処理

ログイン成功後に sessionToken を localStorage へ保存し、一覧画面へ遷移します。

const response = await fetch(apiUrl("/api/login"), {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email, password })
});
const data = await response.json();
localStorage.setItem("cafeNaviSessionToken", data.sessionToken || "");
window.location.href = "index.html";
### 4-2. API呼び出し共通化

AuthorizationヘッダーにBearerトークンを付与します。

const apiFetch = (path, options = {}) => {
  const headers = new Headers(options.headers || {});
  const token = localStorage.getItem("cafeNaviSessionToken") || "";
  if (token) headers.set("Authorization", `Bearer ${token}`);
  return fetch(apiUrl(path), { ...options, headers });
};
